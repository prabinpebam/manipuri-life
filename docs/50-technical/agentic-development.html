Agentic Development Approach

How we use AI agents to narrow the skill gap - covering the technical know-how a small passion team lacks - while humans stay firmly in the loop on design, feel, and truth.

TL;DR

We adopt agent-first, human-in-the-loop development: agents handle the engineering we don't have deep expertise in (engine code, pipelines, tooling, tests), and humans own what agents can't - game feel, art direction, design judgment, and historical truth. This only works if the project is built to be agent-friendly: text-first, typed, tested, and well-documented. Godot fits unusually well - text scenes, a real CLI with --headless, GUT tests for verifiable loops, and community MCP servers that let an agent drive the editor. Guardrails (automated tests, the fact schema, and advisory review) keep agent speed from becoming throwaway work or inaccurate content.

Why this page exists

Becoming a full-stack game developer - engine, graphics, audio, tools, pipelines, ports - is a lot for a small team. The intent here is explicit: use agentic automation to close that skill gap, not to replace the humans. The agent goes where our technical know-how runs out; people stay in control of the decisions that make the game good and true.

The stance: agent-first, human-in-the-loop

"Purely agentic for everything" is the wrong goal. The right one is a clear division of labour, played to each side's strengths.

Agents own (with review)

  • Engine/gameplay code we lack expertise in (GDScript/C#, node wiring, shaders)
  • The fact-to-content pipeline, tooling, and glue
  • Boilerplate, refactors, tests, CI, exports
  • First-draft content generation (gated by schema + review)
  • Explaining and teaching the tech as we go - the skill-gap bridge

Humans own

  • Game feel - pacing, juice, "cozy" - judged by playing
  • Art direction and the overall look
  • Design decisions, scope, and priority calls
  • Historical truth and cultural sensitivity
  • Final review and sign-off on everything shipped

What agents are good and bad at

Being honest about the edges is what keeps the approach safe.

thumb_up

Strong

Text-representable, verifiable work: code, data transforms, tooling, tests, refactors, boilerplate, and content at scale - a natural fit for a data-driven historical game.

thumb_down

Weak

Subjective feel, art direction, long-horizon architecture judgment, and anything requiring playing the game. And they hallucinate - fatal for a game whose value is accuracy unless gated.

Agent-friendliness is an architecture lever

Agent effectiveness isn't fixed - it scales with how we build. The more the project looks like the four things below, the more the agent can do reliably and the less we have to babysit.

Text-first
Scenes, resources, and content as readable text the agent can generate, diff, and review - not opaque binary blobs or GUI-only state.
Typed
Statically-typed code (C#, TypeScript) gives the agent a compiler as a safety net and a tighter feedback loop than dynamically-typed languages.
Tested
Automated tests turn "looks right" into a verifiable pass/fail the agent can iterate against on its own - the single biggest reliability multiplier.
Documented
A large public corpus and clear in-repo docs (like this site) give the agent grounded context instead of guesswork.

The nudge this creates

Prefer C# over GDScript, keep everything text-first and test-covered, and treat the docs as agent context. Choosing the stack for the product comes first - but among good options, the more agent-friendly one wins ties.

Why Godot fits the agentic workflow

The engine we already lean toward (see Tech Stack Research) happens to be one of the most agent-drivable. Its toolchain gives an agent everything it needs to build, run, and check its own work without a human operating the editor.

On the MCP servers - treat as an unproven dependency

Godot MCP servers exist and are active (a Godot-MCP entry sits in the official Asset Library, plus several GitHub projects supporting Claude, Copilot, Cursor, and Gemini), but they are community-maintained, not first-party. Like the console-porting dependency, validate maturity in a POC, pin versions, and never let the workflow hinge on a single one surviving. Even without MCP, the CLI + headless + GUT path already enables most agent work.

Human-in-the-loop guardrails

Agents produce a lot, fast. Without guardrails that speed becomes debt or, worse, confident inaccuracy. Four gates keep it honest.

  1. Tests before trust

    Agent work lands behind automated tests (GUT, pipeline validation). If it isn't verifiable, it isn't done.

  2. Schema-gated content

    Generated content must conform to the fact schema and its confidence model - no free-floating "facts."

  3. Accuracy & sensitivity review

    Anything historical passes the cultural-integrity practice and the 98/2 rule. Agent output is a draft, never a source of truth.

  4. Human sign-off on feel

    A person plays it and decides if it's good. Taste is not delegated.

The one hard rule

For a game whose soul is authenticity, never treat agent-generated history as fact. It is always a starting draft to be verified against real sources and the advisory circle. Convenience must not become fabrication.

How this fits How We Work

This slots cleanly into the agile approach: agents make the parallel tracks move faster, and the decision/dependency tracker keeps them from racing ahead of settled choices. Because agents amplify output, the agile principle of reducing throwaway work matters more, not less - which is exactly what the guardrails above enforce.

First POC - does this actually work?

Rather than assume, we test the workflow itself as an early, low-cost experiment.

  1. Agent-drives-Godot slice

    Point an agent at a small Godot + C# project and have it build one isometric scene, headless, with GUT tests - measuring how far it gets unaided.

  2. MCP trial

    Wire one community Godot MCP server to the agent and evaluate editor-driving reliability and maintenance risk.

  3. Pipeline slice

    Have the agent build the SQLite → validated JSON → in-engine content path end to end, gated by the fact schema.

  4. Judge the loop

    Assess: how much did the agent close the skill gap, how tight was the feedback loop, and where did humans have to step in?

Risks & caveats

  • Hallucination - especially for historical content; mitigated by schema + advisory review.
  • Over-reliance - the team should still understand the system enough to review it; agents assist judgment, not replace it.
  • Immature tooling - community MCP servers may lag Godot releases; keep the CLI/headless path as the dependable baseline.
  • Feel can't be automated - budget human time for the play-and-tune work no agent can do.
