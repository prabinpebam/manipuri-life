POC Plan

How we start building - keeping this repo clean for documentation, spinning up a separate throwaway repo for proof-of-concepts, and running a sequence of tiny experiments that each answer one question and feed a verdict back into the tracker.

TL;DR

This repo stays documentation and tracking only. Code lives in a separate manipuri-life-poc repo (one repo, a folder per experiment) so POCs stay disposable and easy to compare. Before starting we need a short toolchain (Godot 4 .NET, .NET SDK, Git + Git LFS, GitHub, VS Code, a pixel editor, CC0 placeholder assets, GUT) - almost all free. Then we run small, timeboxed POCs, each answering one question (render, input, tilemap, data pipeline, dialogue, save, console, agent-driving), ending in a written keep / adapt / drop verdict that updates the decision tracker. Winners later graduate into a real manipuri-life-game repo.

This documentation repo stays clean

The manipuri-life repo you're reading is the source of truth for decisions and tracking - not a code repo. Experiments, prototypes, and engine projects live elsewhere so this stays a stable, reviewable record. POC results flow back here as verdicts; POC code does not.

Repo strategy - where code goes

Two sane options for the POC code. For early, disposable experiments, one repo with a folder per POC is the least overhead and makes cross-comparison easy.

folder

A · One POC repo, folders per experiment Recommended

manipuri-life-poc with poc-01-iso-render/, poc-02-input/, etc., plus an index README. Low overhead, easy to browse side by side, one place for shared placeholder assets. Best while things are throwaway.

account_tree

B · One repo per POC

Separate small repos (poc-iso-render, …). Cleaner isolation and independent history, but more repos to create and juggle. Worth it only if a POC grows into something real.

Proposed repos

manipuri-life (this one) - docs & tracking.  ·  manipuri-life-poc - all early experiments, disposable.  ·  manipuri-life-game - created later, once a vertical slice proves the stack, seeded from the POC winners.

Placeholder assets are not final art

Use CC0 packs and colored blocks to prove mechanics and feel - never ship them. Track any third-party asset's licence and attribution in the POC's README. Our own art explorations are reference, not drop-in assets. Keeping placeholders obviously temporary avoids mistaking a prototype for the game.

How each POC works - the rules

POCs are experiments, not products. A few rules keep them cheap and useful.

  1. One question each

    Every POC exists to answer a single question (e.g. "does the cozy iso look hold with a day/night cycle?"). If it answers two, split it.

  2. Timeboxed & throwaway

    Small by design; the code is allowed to be ugly and disposable. We're buying learning, not shippable code.

  3. Placeholder-first

    Stand-in assets and stub data. No real art or content until a mechanic is proven worth it.

  4. Ends in a verdict

    Each POC's README records a keep / adapt / drop verdict and a one-line lesson, which updates the decision & dependency tracker back here.

The POC sequence

Ordered from "does the toolchain even work" outward. Each is independent enough to start where inspired, but this order minimises rework. POC 0 comes first; the rest can parallelise.

#POCQuestion it answersSuccess looks like
0Environment & "hello Godot"Can we build/run/test from the CLI, and can an agent drive it?godot --headless runs a scene and a passing GUT test; committed with LFS.
1Isometric render + day/nightDoes the cozy pixel look hold, with Y-sort and a lighting cycle?A small iso scene reads well at morning/noon/night with correct sorting.
2Movement + input abstractionCan one action-set drive keyboard, gamepad, and touch?A placeholder character moves identically across all three inputs.
3Tilemap + a tiny villageWhat's the level-building pipeline (LDtk / Tiled / Godot TileMap)?A walkable village block with collision, authored quickly and re-importable.
4Data pipeline sliceCan the fact database drive in-game content?SQLite → validated JSON → NPCs/crops spawned from data, schema-checked.
5Dialogue sliceDoes the dialogue runtime fit relationship-led play?One Yarn Spinner conversation with a branching choice, in-scene.
6Save / load sliceCan we persist and migrate state cleanly?Position + a relationship value survive quit/reload via versioned save.
7Console feasibility & MCPIs the console path real, and can an agent drive the editor?Confirmed current W4 terms; a Godot MCP server creates/edits a node reliably.

Toward a vertical slice

Once the individual POCs pass, a small vertical slice stitches the winners together - walk a data-driven village, talk to one neighbour, do one seasonal task, save and reload. That slice is the trigger to spin up manipuri-life-game and (only then) engage console porting.

How POC results feed the docs

The loop that keeps this repo the source of truth:

Per-POC README
Goal, the question, what was tried, the verdict (keep/adapt/drop), and one lesson. Lives in the POC repo.
Decision tracker update
Verdicts flip entries in How We Work's decision & dependency tracker (e.g. engine, tilemap tool, dialogue runtime move from "open" to "decided").
Tech pages updated
Findings refine Tech Stack Research and Agentic Development - proposals become decisions with evidence.

First concrete step

  1. Create the POC repo

    New manipuri-life-poc repo with a Godot .gitignore, Git LFS for binaries, and an index README.

  2. Install the toolchain

    Godot 4 (.NET), the .NET SDK, Git LFS, and the GUT addon.

  3. Run POC 0

    A one-scene project that runs headless and passes a GUT test - proving the build/test loop and, optionally, an agent driving it.

  4. Record the verdict

    Write the POC 0 README and update the decision tracker here.

Want this scaffolded?

When you're ready, this can be bootstrapped for you: the manipuri-life-poc repo structure, a Godot .gitignore and LFS config, a POC index README, and a POC 0 skeleton with a GUT test - so you start from a running loop rather than a blank folder.
