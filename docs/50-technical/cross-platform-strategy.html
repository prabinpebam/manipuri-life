Cross-Platform Strategy

How we plan to ship on PC, smartphone, and console as one coherent product - a first-class constraint on both design and technical execution, decided now so it never becomes rework.

TL;DR

We treat cross-platform as a P1 constraint from foundation, learning directly from our benchmarks. Four commitments drive everything: a fixed platform matrix; an input-agnostic, mobile-first UI (every action works with touch, gamepad, and keyboard/mouse); resolution independence with true fullscreen; and a portable, cloud-backed, versioned save system. This page splits into design commitments and technical execution so both disciplines own their half.

A constraint, not a phase

Stardew's mobile lag and Japanese Rural Life Adventure's rough PC port both came from porting after the fact. Cross-platform is not a milestone near launch; it is a rule that shapes the engine choice, the UI, the input model, and the save format from the first prototype.

Why PC plus mobile at launch

They are the two hardest input models to reconcile - precise mouse and imprecise touch. If the game feels right on both from day one, controller and console are comparatively easy. Designing for the extremes first is the cheapest path to all platforms.

Part A - Design commitments

These are obligations on Game Design, derived from the benchmarks.

  1. Input-agnostic interaction

    Every action is expressible with touch, gamepad, and keyboard/mouse. No hover-only tooltips, no right-click-only actions, no drag chains that a thumb cannot perform, no tiny targets.

  2. Mobile-first, one-handed core

    The daily loop is playable one-handed on a phone. Menus are reachable in the thumb zone; the most frequent actions cost the fewest taps.

  3. No precision-twitch requirements

    Fishing, minigames, and any combat must be forgiving and optional, never demanding pixel timing that dies on touch - the exact Stardew mobile complaint.

  4. Session shape for every device

    Support a two-minute phone session (open, do one meaningful thing, save-anywhere, close) and a long desktop session equally well.

  5. Readable at phone size

    Type scale, contrast, and safe-area margins are validated on a small handset first, then allowed to breathe on larger screens.

  6. Full feature parity

    No platform-exclusive core system. What a player can do is identical everywhere; only input and presentation adapt.

  7. Multiplayer scope decided now

    Proposed: single-player at launch, with co-op treated as an explicit, netcode-shaping decision rather than a later bolt-on. Decision needed

Cozy and cross-platform pull the same way

Simplifying interactions so a thumb can perform them also removes friction - which is our core cozy goal. The platform constraint and the design pillar reinforce each other; we are not trading one for the other.

Part B - Technical execution

These are obligations on the engine, architecture, and pipeline.

memory

Engine selection criteria

First-class export to PC, iOS, Android, and Switch; a strong 2D/pixel pipeline; and built-in input abstraction. Candidates to evaluate include Godot and Unity, weighed against console-porting realities. Decision needed

touch_app

Input abstraction layer

Bind game actions, not devices. Touch, gamepad, and keyboard/mouse all map onto one action set, with per-device UI hints swapped at runtime.

aspect_ratio

Resolution independence

A virtual canvas with dynamic UI scaling, safe-area handling, and true fullscreen across aspect ratios - engineered to avoid Japanese Rural Life Adventure's scaling and letterboxing complaints.

speed

Performance budget

The lowest-end target phone sets the performance ceiling. We profile on a modest handset continuously, not only on a dev PC.

save

Portable save system

An engine-neutral, serialized, versioned save with migration paths and cloud backup - and a cross-save design decided early rather than retrofitted.

sync

Parity release pipeline

CI builds every target from one source so features land together, preventing the mobile-lag resentment Stardew still carries.

sports_esports

Console realities

Certification, devkits, memory limits, and controller requirements are planned early - likely with a porting partner - not discovered near launch.

cable

Per-platform plumbing

Achievements, cloud saves, and store integration abstracted behind one interface so each storefront is a thin adapter.

Sequencing and risk

  1. Decide the matrix and engine at Foundation

    Platform tiers, lead mobile OS, and engine are ratified before the vertical slice. Everything downstream depends on them.

  2. Prototype input first

    Before any content, prove the daily loop feels right on a phone, a gamepad, and keyboard/mouse using the same UI. If it does not, fix it before scaling.

  3. Test on a low-end phone throughout

    Continuous handset testing keeps the performance and readability budgets honest.

  4. Keep parity via CI

    All targets build from one branch from the start, so no platform silently drifts behind.
